Managed Agents 的可扩展架构
把大脑和双手解耦

Harness 会随模型进步而过时。Managed Agents 围绕稳定接口构建——让 harness 内部实现自由演变，而接口契约保持不变。

作者：Lance Martin · Gabe Cemaj · Michael Cohen

一个 harness 的每个组件，都在隐式地表达"模型自身还做不到这件事"这一假设。随着模型进步，这些假设会迅速过时。举个具体的例子：Claude Sonnet 4.5 会出现 context anxiety，在接近上下文限制时提前结束任务。当我们切换到 Claude Opus 4.5 后，这个行为彻底消失，原本围绕 context reset 搭建的大量 harness 代码瞬间变成了死重。这正是我们把 Managed Agents 建立在稳定接口之上的原因——让 harness 内部的实现尽情变化，而接口本身不动。

Managed Agents 是 Anthropic 托管的 agent 服务，专为长时程任务设计。它的核心设计思路是：对 session、harness 和 sandbox 三个组件进行"虚拟化"——每个组件都有简单、稳定的接口，可以独立失败、独立替换，彼此不产生硬依赖。

不要养宠物

图：早期单体容器架构——所有组件耦合在同一个容器里

最初，我们把所有 agent 组件都部署在一个容器里，形成了一个需要精心呵护的"宠物"——有名字、有状态、一旦挂掉就麻烦大了。这种架构带来了几个连锁问题：

🐾 宠物式架构的问题

  • 容器崩溃 → session 数据丢失
  • WebSocket event stream 无法定位故障来源（harness？网络？容器？）
  • 调试需要进入容器内部，但容器里有用户数据，存在安全隐患
  • 连接客户 VPC 需要网络 peering，或把 harness 移到容器外部——这是写死在架构里的假设

🐄 牲畜式架构的目标

  • 容器崩溃只是一次工具调用错误，可自动重试
  • 故障边界清晰，每个组件独立可观测
  • harness 在容器外运行，安全边界明确
  • brain 和 hands 各自独立扩展，互不依赖

解决方案借鉴了运维领域"Pets vs. Cattle"的思想：把"宠物"变成"牲畜"——把有状态、难替换的单体，拆分为无状态、可独立扩展的组件，通过简洁接口连接。

三组核心接口

Hands 接口 — 统一工具调用

Harness 通过一个统一接口调用 sandbox，将容器视为普通工具：

execute(name, input) → string

容器崩溃时，返回的工具错误会直接传给 Claude，然后 harness 通过以下接口自动重新 provision：

provision({ resources })

失败的容器不需要人工干预，整个过程对上层完全透明。

Brain 恢复接口 — Harness 崩溃无损

Session log 存在 harness 之外，harness 崩溃不会丢失任何状态。新的 harness 实例启动后：

接口作用
wake(sessionId) 唤醒一个已有 session，从上次记录的 event 位置继续
getSession(id) 获取 session 的完整事件历史
emitEvent(id, event) 在运行过程中向 session 写入持久化事件记录
getEvents() 从 session 读取事件（支持位置切片）

安全边界 — 凭据与沙箱分离

解耦之后，不可信代码和用户凭据不再共处同一个 sandbox：

  • Git 操作：仓库 token 只用于初始化 clone 和配置 remote，agent 本身不直接接触凭据。
  • OAuth token：存储在安全 vault 中，由 proxy 代为调用，session 关联的凭据对 agent 不可见。
  • 自定义工具：通过 MCP proxy 路由，harness 无需感知凭据的存储位置。
图：Brain / Session / Hands 三组件解耦后的架构示意

Session ≠ Claude 的 Context Window

长时程任务的执行时间往往超出 Claude 的 context window 容量，这就需要决策：哪些信息保留，哪些丢弃？

在新架构中，session 是一个独立于 context window 的持久化上下文对象。Harness 可以通过 getEvents() 按位置切片读取事件，让 brain 具备以下能力：

  • 续读：从上次中断的位置继续，而非从头重放整个历史。
  • 回溯：回到特定时间点重新审视之前的 context。
  • 变换：Harness 可以在把事件传入 Claude context window 之前对其进行处理——包括 cache 优化和 context engineering。
图：Session（持久化存储）与 Claude context window（临时工作区）的关系
关键设计决策：将持久化存储（session）和上下文管理（harness）分离，意味着我们不需要对"未来的 context engineering 会是什么样"做任何假设。当更好的 context engineering 策略出现时，只需更新 harness 的处理逻辑，session 接口保持不变。

多大脑，多双手

Many Brains — 性能突破

解耦带来了一个意外的性能红利。原来，brain 和 hands 必须共处同一个容器，这意味着每个 session 启动时都要承担完整的 provision 开销：clone 仓库、启动进程、拉取 pending events——无论这个 session 是否真的需要用到 sandbox。

新架构引入了懒加载 provision：只有当 brain 通过 execute(name, input) → string 显式请求时，sandbox 才会启动。对于不需要 sandbox 的 session，推理可以立即开始。

这带来了显著的延迟降低：

⚡ p50 TTFT 下降约 60% 🚀 p95 TTFT 下降超过 90%

TTFT（Time-to-First-Token）是用户感知最直接的延迟指标——从发出请求到看到第一个 token 输出的时间。p95 下降 90% 意味着对于尾部（最慢的那些）请求，体验改善尤为显著。

图：解耦前后 TTFT 对比，p50 和 p95 均大幅下降

扩展能力也随之改善：brain（harness）本身是无状态的，只需水平复制即可扩容，按需连接对应的 hands，无需担心状态同步问题。

此外，解耦还解决了一个客户真实需求：此前，将 Claude 连接到客户私有 VPC 需要网络 peering，或者将 harness 搬到容器之外——这是写死在架构里的假设。新架构中，brain 和 hands 天然就在不同的网络位置，连接私有 VPC 只是一个配置问题。

Many Hands — 灵活的执行层

Brain 现在可以感知多个执行环境，并主动调度工作——这是一个需要足够模型能力才能胜任的认知任务。早期模型无法可靠地完成这个任务，所以当时只能用单容器方案。

统一的 execute(name, input) → string 接口让 harness 对"hands 是什么"保持无感：

Hands 类型说明
容器 (Container)标准沙箱执行环境，用于代码运行和文件操作
MCP Server任意 MCP 协议兼容的工具服务
手机 / 模拟器移动端测试和自动化场景
自定义工具内部实现的专用执行环境
其他 AgentBrain 可以将 hands 委托给另一个 agent（sub-agent）
Brain 委托 Hands 给另一个 Brain：在这套架构里，一个 agent 可以将工具执行委托给另一个 agent——即 sub-agent 模式。这使得 multi-agent 编排成为架构的自然延伸，而非特殊处理。

结语

这个问题和计算机科学中的经典难题如出一辙：如何为"尚未被想到的程序"设计系统？操作系统的答案是对硬件进行虚拟化——process、file 这些抽象比底层实现活得更久。Managed Agents 把同样的思路应用到 agent 架构上，将 session、harness 和 sandbox 都视为可虚拟化的组件。

Managed Agents 本质上是一个 "meta-harness"——它不对具体的 harness 实现表达任何偏好，而是支持各类不同的实现：Claude Code、任务专用 agent，以及随着模型能力演进而出现的各种新形态。

这套架构做出了明确的承诺：围绕状态操作和计算的通用接口契约保持稳定。它不假设 brain 和 hands 的数量，不假设它们的位置，同时保证长时程操作的可靠性和安全性。

技术要点总结

本节梳理文章中所有关键技术点，方便快速回顾。

核心问题与动机

  • Harness 的每个组件都编码了关于"模型能力边界"的假设，这些假设随模型进步迅速过时。
  • 早期单体容器架构（brain + hands + session 全在一个容器）存在：状态丢失、故障难排查、安全边界模糊、网络依赖硬编码等问题。
  • context anxiety（Claude Sonnet 4.5 在接近 context 限制时过早结束任务）是推动架构演进的直接导火索。

三组件虚拟化架构

  • Brain = Claude + Harness（决策层，无状态）
  • Session = 只追加的 event log（持久化状态，独立于 brain 和 hands）
  • Hands = Sandbox / 工具执行层（容器、MCP server、手机、sub-agent 等）
  • 三者通过简洁接口连接，可独立失败、独立替换、独立扩展。

核心接口设计

  • execute(name, input) → string：统一 hands 调用接口，harness 对 hands 的具体实现无感。
  • provision({resources})：容器崩溃后自动重新 provision，无需人工干预。
  • wake(sessionId)：harness 崩溃重启后，从上次 event 位置无缝继续。
  • getSession(id) / emitEvent(id, event)：session 的读写接口，保证状态持久化。
  • getEvents()：支持位置切片读取，harness 可按需变换事件再传入 Claude context window。

Session vs. Context Window 的分离

  • Session 是持久化存储（append-only event log），context window 是 Claude 的临时工作区。
  • Harness 通过 getEvents() 决定哪些历史事件进入 context window，支持续读、回溯、变换（cache 优化、context engineering）。
  • 这种分离让 context engineering 策略可以独立演进，不影响 session 接口。

性能收益

  • 懒加载 provision：只有 brain 显式调用 execute() 时，sandbox 才启动。不需要 sandbox 的 session 立即开始推理。
  • p50 TTFT 下降约 60%，p95 TTFT 下降超过 90%。
  • Brain（harness）无状态，可水平扩展，按需连接对应的 hands。

安全边界改进

  • 不可信代码与用户凭据不再共处同一 sandbox。
  • Git token 只用于初始化 clone，agent 不直接接触凭据。
  • OAuth token 存储在 vault 中，通过 MCP proxy 代理调用。

Many Brains, Many Hands 的扩展能力

  • Brain 可感知并调度多个执行环境（多个 hands），依赖模型自身的足够推理能力。
  • 统一 execute(name, input) → string 接口支持容器、MCP server、手机/模拟器、sub-agent 等任意 hands 类型。
  • 连接客户私有 VPC 从"架构级假设"降级为普通配置问题。
  • Brain 可将 hands 委托给另一个 agent，实现自然的 multi-agent 编排。

设计哲学

  • 类比操作系统对硬件的虚拟化：用稳定抽象（process、file）屏蔽底层实现变化。Managed Agents 用稳定接口屏蔽 harness 实现的变化。
  • Managed Agents 是一个 meta-harness：不对具体 harness 实现表达偏好，支持 Claude Code、任务专用 agent 等各类实现并存。
  • 架构承诺稳定的接口契约，不假设 brain 和 hands 的数量与位置。
