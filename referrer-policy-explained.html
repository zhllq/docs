1. Fundamental Concepts

🌐 URL Structure

Before diving into referrer policies, let's understand URL components:

https://www.example.com:443/products/item?id=123&ref=home
  • Scheme Protocol used (http, https)
  • Host Domain name or IP address
  • Port Network port (often implicit: 80 for HTTP, 443 for HTTPS)
  • Path Resource location on the server
  • Query Additional parameters (may contain sensitive data!)

2. The Referer Header

📤 What is the Referer Header?

The Referer (note: historically misspelled) is an HTTP header that browsers automatically send with requests. It tells the destination server where the user came from.

Example Scenario:
You're on https://blog.example.com/article/secret-tips and click a link to https://analytics.com.

The browser sends this header to analytics.com: 
Referer: https://blog.example.com/article/secret-tips

Why Referer Matters

✅ Useful For
  • Analytics (traffic sources)
  • CSRF protection
  • Access control decisions
  • Debugging broken links
⚠️ Privacy Risks
  • Leaks sensitive URL paths
  • Exposes query parameters (tokens, IDs)
  • Reveals browsing patterns
  • Can leak internal URLs

3. Understanding Origin

🎯 What is an Origin?

An origin is defined by three components:

Component Example
Scheme (Protocol) https
Host (Domain) www.example.com
Port 443 (default for HTTPS)

Origin = Scheme + Host + Port

Example: https://www.example.com:443 → Origin is https://www.example.com

Origin Examples

Full URL Origin
https://example.com/page?q=test https://example.com
http://api.example.com:8080/data http://api.example.com:8080
https://shop.example.com/cart https://shop.example.com

Note: Path and query string are NOT part of the origin!

4. Same-Origin vs Cross-Origin

🔄 Same-Origin

Two URLs are same-origin if they have the same scheme, host, AND port.

Base URL: https://example.com/page1

Target URL Same Origin? Reason
https://example.com/page2 ✓ Yes Same scheme, host, port
https://example.com/dir/page ✓ Yes Path doesn't matter
http://example.com/page ✗ No Different scheme (http vs https)
https://api.example.com/page ✗ No Different host (subdomain)
https://example.com:8443/page ✗ No Different port
https://other.com/page ✗ No Different host
🌍 Cross-Origin

Any request to a different origin is considered cross-origin. This triggers various browser security mechanisms:

  • CORS (Cross-Origin Resource Sharing) - controls API access
  • Referrer Policy - controls what URL info is sent
  • Cookie restrictions - SameSite attribute

5. Referrer Policy Overview

📋 Available Policies
Policy Behavior
no-referrer Never send Referer header
no-referrer-when-downgrade Send full URL, but nothing on HTTPS→HTTP
origin Always send origin only (no path)
origin-when-cross-origin Full URL for same-origin, origin only for cross-origin
same-origin Full URL for same-origin, nothing for cross-origin
strict-origin Origin only, nothing on downgrade
strict-origin-when-cross-origin Default policy (see below)
unsafe-url Always send full URL (not recommended)

6. strict-origin-when-cross-origin Explained

⭐ The Default Browser Policy

This is the default referrer policy in modern browsers (Chrome 85+, Firefox 87+, Safari 14+). It provides a good balance between functionality and privacy.

The Three Rules

1️⃣ Same-Origin Requests

Send the full URL (including path and query string)

https://example.com/secret/page?token=abc
→
https://example.com/other
=
Referer: https://example.com/secret/page?token=abc

Rationale: Same site, same trust level. Full analytics data is useful.

2️⃣ Cross-Origin (Same Security Level)

Send origin only (scheme + host, no path or query)

https://example.com/secret/page?token=abc
→
https://analytics.com/track
=
Referer: https://example.com

Rationale: Third party only needs to know where traffic came from, not specific pages.

3️⃣ Security Downgrade (HTTPS → HTTP)

Send nothing (empty Referer header)

https://example.com/page
→
http://insecure.com/
=
Referer: (empty)

Rationale: Never leak HTTPS URLs to insecure HTTP destinations!

7. Practical Examples

Example A: Internal Navigation (Same-Origin)

Scenario: User navigates within the same website

https://shop.example.com/cart?items=3
→ clicks link →
https://shop.example.com/checkout

Referer sent: https://shop.example.com/cart?items=3

Result: Full URL sent - The checkout page can see exactly where the user came from.

Example B: External Link (Cross-Origin, HTTPS→HTTPS)

Scenario: User clicks an external link to a different website

https://blog.example.com/post/my-secret-draft?preview=true
→ clicks link →
https://twitter.com/share

Referer sent: https://blog.example.com

Result: Origin only - Twitter knows traffic came from blog.example.com but NOT the specific draft URL or preview token.

Example C: Loading External Resources

Scenario: Page loads an image from a CDN

<!-- On page: https://app.example.com/dashboard/user/12345 -->
<img src="https://cdn.images.com/logo.png">

Referer sent: https://app.example.com

Result: Origin only - CDN doesn't see user ID in the URL path.

Example D: Security Downgrade

Scenario: HTTPS page links to HTTP resource (bad practice!)

https://secure-bank.com/account/transfer
→ loads image →
http://old-tracker.com/pixel.gif

Referer sent: (nothing)

Result: Empty - Protects against leaking secure URLs over insecure connections.

Example E: API Fetch Request

Scenario: JavaScript makes an API call

// On page: https://app.example.com/users/settings?tab=security

fetch('https://api.example.com/user/profile')
  .then(response => response.json());

Referer sent: https://app.example.com

Why? app.example.com ≠ api.example.com (different subdomain = different origin = cross-origin request)

Example F: Subdomain Navigation

Scenario: User moves from main site to documentation subdomain

https://www.example.com/pricing?plan=pro
→ clicks →
https://docs.example.com/getting-started

Referer sent: https://www.example.com

Result: Origin only - Subdomains are different origins!

8. How to Implement

🔧 Setting Referrer Policy

Method 1: HTTP Header (Recommended)

Referrer-Policy: strict-origin-when-cross-origin

Method 2: HTML Meta Tag

<meta name="referrer" content="strict-origin-when-cross-origin">

Method 3: Per-Element Attribute

<a href="https://example.com" referrerpolicy="no-referrer">Link</a>

<img src="image.jpg" referrerpolicy="origin">

Method 4: Fetch API

fetch('https://api.example.com/data', {
  referrerPolicy: 'strict-origin-when-cross-origin'
});
💡 Best Practices
  • Use strict-origin-when-cross-origin as your default (it already is in modern browsers)
  • Use no-referrer for highly sensitive pages (password reset, etc.)
  • Never use unsafe-url unless absolutely necessary
  • Don't put sensitive data in URLs (tokens, session IDs, etc.)
  • Always use HTTPS to prevent unintentional referrer stripping
📝 Summary

strict-origin-when-cross-origin is the modern default that:

  • ✅ Preserves full referrer information within your own site (useful for analytics)
  • ✅ Protects user privacy by sending only the origin to third parties
  • ✅ Prevents any data leakage when downgrading from HTTPS to HTTP

It's the "Goldilocks" policy—not too strict, not too permissive, just right for most use cases.
