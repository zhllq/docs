Claude Code 最佳实践
Best Practices for Claude Code

原文链接: https://code.claude.com/docs/en/best-practices
从环境配置到多会话并行，本文汇总了高效使用 Claude Code 的技巧与模式。

Claude Code 是一个agent 编程环境。它不是等你提问的聊天机器人——它能读文件、跑命令、改代码，你可以在旁边看，随时纠偏，也可以直接走开让它自主完成。

这改变了你的工作方式。你不再自己写代码然后让 Claude 审查，而是描述你想要什么，Claude 来探索、规划、实现。

但自主性也有学习曲线。Claude 在特定约束下工作，你需要理解这些约束。

本指南涵盖了 Anthropic 内部团队和外部工程师在各种代码库、语言、环境中验证过的有效模式。关于agent 循环的底层原理，参见 How Claude Code works。

多数最佳实践都基于一个约束：Claude 的 context window 填充速度很快，填满后性能下降。

Context window 装载你的整个对话：每条消息、Claude 读的每个文件、每条命令输出。一次调试或代码探索就可能消耗数万 token。

这很重要——LLM 在 context 填满时性能下降。Context window 快满时，Claude 可能「遗忘」早期指令或犯更多错。Context window 是你需要管理的最重要资源。要了解会话中 context 的实际填充情况，参见交互式演示。可用自定义状态栏持续跟踪 context 用量，也可参考减少 token 用量的策略。

让 Claude 有办法验证自己的工作

提供测试、截图或预期输出，让 Claude 能自我检查。这是你能做的最高杠杆的事。

Claude 在能自行验证时表现显著提升——跑测试、比对截图、校验输出。

没有明确的成功标准，它可能产出看起来正确但实际不工作的东西。你成了唯一的反馈回路，每个错误都需要你的注意力。

策略改进前改进后
提供验证标准 "实现一个验证邮箱的函数" "写一个 validateEmail 函数。测试用例：user@example.com 返回 true，invalid 返回 false，user@.com 返回 false。实现后跑测试。"
视觉验证 UI 变更 "把仪表盘做得好看点" "[粘贴截图] 实现这个设计。截图比对结果与原图，列出差异并修复。"
定位根因而非症状 "构建失败了" "构建失败，错误信息：[粘贴错误]。修复它并验证构建成功。解决根因，不要压制错误。"

UI 变更可通过 Claude in Chrome 扩展验证——它在浏览器中打开新标签页，测试 UI 并迭代到代码正常运行。

验证手段也可以是测试套件、linter、或检查输出的 Bash 命令。在验证手段上多投入。

先探索，再规划，最后写代码

把调研和规划从实现中分离出来，避免解决错误的问题。

让 Claude 直接动手写代码，可能写出解决错误问题的代码。用 Plan Mode 将探索与执行分开。

推荐的工作流分四个阶段：

1. 探索

进入 Plan Mode。Claude 读文件、回答问题，但不做任何修改。

read /src/auth and understand how we handle sessions and login.
also look at how we manage environment variables for secrets.
2. 规划

让 Claude 创建详细实现方案。

I want to add Google OAuth. What files need to change?
What's the session flow? Create a plan.

按 Ctrl+G 可在编辑器中打开方案直接编辑，然后再让 Claude 执行。

3. 实现

切回 Normal Mode，让 Claude 按方案编码并验证。

implement the OAuth flow from your plan. write tests for the
callback handler, run the test suite and fix any failures.
4. 提交

让 Claude 用描述性消息提交并创建 PR。

commit with a descriptive message and open a PR

Plan Mode 有用，但也有开销。

对于范围明确、改动很小的任务（修 typo、加一行日志、重命名变量），直接让 Claude 做即可。

规划在以下情况最有价值：不确定方案、变更涉及多个文件、或对要改的代码不熟悉。如果你能一句话描述 diff，跳过规划。

在 prompt 中提供具体上下文

指令越精确，你需要的修正越少。

Claude 能推断意图，但读不了你的心。引用具体文件、提及约束、指向示例模式。

策略改进前改进后
界定任务范围。指定文件、场景和测试偏好。 "给 foo.py 加测试" "为 foo.py 写测试，覆盖用户已登出的边界场景。不要用 mock。"
指向信息源。引导 Claude 到能回答问题的源头。 "为什么 ExecutionFactory 的 API 这么怪？" "查看 ExecutionFactory 的 git 历史，总结其 API 是如何演变至今的"
引用已有模式。指向代码库中的现有模式。 "加一个日历组件" "看一下首页现有 widget 的实现方式来理解模式。HotDogWidget.php 是个好例子。按相同模式实现一个新的日历 widget，支持选择月份和前后翻页选年。从零构建，不引入代码库中未使用的库。"
描述症状。提供症状、可能位置、以及「修好」的定义。 "修复登录 bug" "用户报告 session 超时后登录失败。检查 src/auth/ 中的认证流程，尤其是 token 刷新。写一个能复现问题的失败测试，然后修复。"

模糊 prompt 在探索阶段也有价值——当你能承受纠偏成本时。像 "你觉得这个文件有什么可以改进的？" 这类提问可以发现你未曾想到的问题。

提供丰富内容

用 @ 引用文件、粘贴截图/图片、或管道传入数据。

你可以通过多种方式向 Claude 提供丰富数据：

  • 用 @ 引用文件，而非描述代码在哪。Claude 会在回复前读取文件。
  • 直接粘贴图片。复制粘贴或拖拽图片到 prompt 中。
  • 给 URL。用于文档和 API 参考。用 /permissions 将常用域名加入白名单。
  • 管道传入数据。运行 cat error.log | claude 将文件内容直接发送。
  • 让 Claude 自己获取所需内容。告诉 Claude 用 Bash 命令、MCP 工具或读文件来拉取上下文。

配置你的环境

几步配置即可让 Claude Code 在所有会话中显著更高效。关于扩展功能的完整概览及使用时机，参见 Extend Claude Code。

写好 CLAUDE.md

运行 /init 基于当前项目结构生成初始 CLAUDE.md，然后逐步完善。

CLAUDE.md 是一个特殊文件，Claude 在每次对话开始时读取。在其中写入 Bash 命令、代码风格和工作流规则。它为 Claude 提供无法从代码推断的持久上下文。

/init 命令会分析你的代码库，检测构建系统、测试框架和代码模式，为你提供可靠的起点。

CLAUDE.md 没有固定格式，保持简短、可读即可。示例：

# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Be sure to typecheck when you're done making a series of code changes
- Prefer running single tests, and not the whole test suite, for performance

CLAUDE.md 每次会话都会加载，所以只放通用内容。特定领域知识或偶尔用到的工作流，用 skills 代替——Claude 按需加载，不会撑大每次对话。

保持精简。对每一行问：「删掉这行会导致 Claude 犯错吗？」 如果不会，删。臃肿的 CLAUDE.md 会导致 Claude 忽略你的真正指令！

应该包含不应包含
Claude 猜不到的 Bash 命令Claude 读代码就能推断的东西
与默认不同的代码风格规则Claude 已知的标准语言惯例
测试说明和首选测试运行器详细 API 文档（链接过去即可）
仓库规范（分支命名、PR 惯例）频繁变更的信息
项目特定的架构决策冗长的解释或教程
开发环境怪癖（必需的环境变量）逐文件描述代码库
常见陷阱或非显而易见的行为"写干净的代码"之类不言自明的做法

如果 Claude 在有规则的情况下仍反复犯错，文件可能太长、规则被淹没了。如果 Claude 问了 CLAUDE.md 中已回答的问题，措辞可能有歧义。把 CLAUDE.md 当代码对待：出问题时审查它，定期修剪，通过观察 Claude 行为是否改变来测试修改效果。

可以用强调语（如「IMPORTANT」或「YOU MUST」）提升规则的遵守度。将 CLAUDE.md 提交到 git，让团队共同维护。此文件的价值随时间复利增长。

CLAUDE.md 可以用 @path/to/import 语法导入其他文件：

See @README.md for project overview and @package.json for available npm commands.

# Additional Instructions
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.claude/my-project-instructions.md

CLAUDE.md 可放在多个位置：

  • 主目录 (~/.claude/CLAUDE.md)：适用于所有 Claude 会话
  • 项目根目录 (./CLAUDE.md)：提交到 git，与团队共享
  • 项目根目录 (./CLAUDE.local.md)：个人项目笔记；将此文件加入 .gitignore
  • 父目录：适用于 monorepo（译者注：指单一仓库管理多个项目的代码组织方式），root/CLAUDE.md 和 root/foo/CLAUDE.md 均自动加载
  • 子目录：Claude 处理该目录文件时按需加载子目录的 CLAUDE.md

配置权限

用 auto mode 让分类器（译者注：指自动审查命令安全性的独立模型）处理审批，用 /permissions 将特定命令加白，或用 /sandbox 做 OS 级隔离。三者都能减少中断，同时保持控制。

默认情况下，Claude Code 对可能修改系统的操作（文件写入、Bash 命令、MCP 工具等）都请求权限。安全但烦人——批准了十次之后你其实已不在认真看了。三种方式减少中断：

  • Auto mode：独立分类器模型审查命令，仅阻止看起来有风险的操作——权限升级、未知基础设施、恶意内容驱动的操作。适合你信任任务大方向但不想逐步点击批准的场景。
  • 权限白名单：允许你已知安全的特定工具，如 npm run lint 或 git commit。
  • 沙箱：启用 OS 级隔离，限制文件系统和网络访问，让 Claude 在限定边界内更自由地工作。

详见 permission modes、permission rules 和 sandboxing。

使用 CLI 工具

告诉 Claude Code 在与外部服务交互时使用 CLI 工具，如 gh、aws、gcloud、sentry-cli。

CLI 工具是与外部服务交互时最节省 context 的方式。用 GitHub 就装 gh CLI——Claude 会用它创建 issue、开 PR、读评论。没有 gh，Claude 仍可用 GitHub API，但未认证请求容易触发速率限制。

Claude 也擅长学习它不认识的 CLI 工具。试试：「用 'foo-cli-tool --help' 了解 foo 工具，然后用它解决 A、B、C。」

连接 MCP 服务器

运行 claude mcp add 连接 Notion、Figma、数据库等外部工具。

通过 MCP 服务器（译者注：Model Context Protocol，模型上下文协议，允许 Claude 与外部工具交互的协议），你可以让 Claude 从 issue 追踪器实现功能、查询数据库、分析监控数据、集成 Figma 设计、自动化工作流。

设置 hooks

用 hooks（译者注：此处指 Claude Code 的钩子系统，在 Claude 工作流特定节点自动执行脚本，非 git hooks） 处理必须零例外执行的操作。

Hooks 在 Claude 工作流的特定节点自动运行脚本。与 CLAUDE.md 中的指令（建议性的）不同，hooks 是确定性的，保证操作一定执行。

Claude 可以帮你写 hooks。试试：「写一个 hook，每次编辑文件后运行 eslint」 或 「写一个 hook，阻止写入 migrations 文件夹」。直接编辑 .claude/settings.json 手动配置 hooks，或运行 /hooks 浏览已有配置。

创建 skills

在 .claude/skills/ 中创建 SKILL.md 文件，为 Claude 提供领域知识和可复用工作流。

Skills（译者注：指 Claude Code 的技能系统，通过 SKILL.md 文件定义的可复用知识和工作流）用项目、团队或领域特定的信息扩展 Claude 的知识。Claude 在相关时自动应用，你也可以用 /skill-name 直接调用。

在 .claude/skills/ 下创建包含 SKILL.md 的目录即可：

# .claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
- Version APIs in the URL path (/v1/, /v2/)

Skills 也可以定义你直接调用的可重复工作流：

# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---
Analyze and fix the GitHub issue: $ARGUMENTS.

1. Use `gh issue view` to get the issue details
2. Understand the problem described in the issue
3. Search the codebase for relevant files
4. Implement the necessary changes to fix the issue
5. Write and run tests to verify the fix
6. Ensure code passes linting and type checking
7. Create a descriptive commit message
8. Push and create a PR

运行 /fix-issue 1234 即可调用。对有副作用的工作流使用 disable-model-invocation: true，确保只能手动触发。

创建自定义 subagent

在 .claude/agents/ 中定义专门的助手，Claude 可将隔离任务委派给它们。

Subagent（译者注：指 Claude Code 中运行在独立 context 中、拥有独立工具集的子代理）在自己的 context 中运行，有自己的工具集。适用于需要读取大量文件或需要专注处理而不应干扰主对话的任务。

# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets or credentials in code
- Insecure data handling

Provide specific line references and suggested fixes.

显式告诉 Claude 使用 subagent：「用一个 subagent 审查这段代码的安全问题。」

安装插件

运行 /plugin 浏览市场。Plugin 无需配置即可添加 skills、工具和集成。

Plugin 将 skills、hooks、subagent 和 MCP 服务器打包成一个可安装单元，来自社区和 Anthropic。如果你用的是强类型语言，安装代码智能插件，为 Claude 提供精确的符号导航和编辑后自动错误检测。

关于在 skills、subagent、hooks 和 MCP 间选择的指导，参见 Extend Claude Code。

高效沟通

你与 Claude Code 的沟通方式直接影响结果质量。

问代码库问题

像问高级工程师一样问 Claude。

入职新代码库时，用 Claude Code 学习和探索。你可以问 Claude 那些你会问另一个工程师的问题：

  • 日志系统是怎么工作的？
  • 如何新建一个 API endpoint？
  • foo.rs 第 134 行的 async move { ... } 是什么意思？
  • CustomerOnboardingFlowImpl 处理了哪些边界情况？
  • 第 333 行为什么调用 foo() 而不是 bar()？

这种方式可以有效缩短上手时间，减少对其他工程师的依赖。无需特殊 prompt 技巧，直接提问。

让 Claude 采访你

对于较大的功能，先让 Claude 采访你。从一个简短 prompt 开始，要求 Claude 用 AskUserQuestion 工具采访你。

Claude 会问你可能没想到的问题——技术实现、UI/UX、边界情况、权衡取舍。

I want to build [brief description]. Interview me in detail using the AskUserQuestion tool.

Ask about technical implementation, UI/UX, edge cases, concerns, and tradeoffs. Don't ask obvious questions, dig into the hard parts I might not have considered.

Keep interviewing until we've covered everything, then write a complete spec to SPEC.md.

规格完成后，开一个新会话来执行。新会话有干净的 context，专注于实现，而你有一份可参考的书面规格。

管理你的会话

对话是持久且可逆的。善用这一点！

尽早、频繁地纠偏

一旦发现 Claude 偏离方向，立即纠正。

最好的结果来自紧密的反馈循环。虽然 Claude 偶尔首次就完美解决问题，但快速纠正通常能更快产出更好的方案。

  • Esc：按 Esc 键中途停止 Claude。Context 保留，可以重新引导。
  • Esc + Esc 或 /rewind：双击 Esc 或运行 /rewind 打开回退菜单，恢复对话和代码状态，或从选定消息开始总结。
  • "Undo that"：让 Claude 撤销其更改。
  • /clear：在无关任务之间重置 context。长会话中不相关的 context 会降低性能。

如果你在同一会话中对同一问题纠正了两次以上，context 中已充满失败的尝试。运行 /clear，用更具体的 prompt（融入你学到的东西）重新开始。干净的会话加上更好的 prompt，几乎总是优于累积纠正的长会话。

积极管理 context

在无关任务之间运行 /clear 重置 context。

Claude Code 在接近 context 上限时自动压缩（译者注：指 auto compaction，自动摘要历史对话以释放 context window 空间）对话历史，保留重要代码和决策，同时释放空间。

长会话中，context window 可能充满无关对话、文件内容和命令输出，降低性能甚至分散 Claude 注意力。

  • 任务之间频繁用 /clear 彻底重置 context window
  • 自动压缩触发时，Claude 总结最重要的内容——代码模式、文件状态、关键决策
  • 需要更多控制时，运行 /compact <instructions>，如 /compact Focus on the API changes
  • 只压缩部分对话：用 Esc + Esc 或 /rewind，选择一个消息检查点，选 Summarize from here。从该点起的消息被浓缩，之前的 context 保持不变。
  • 在 CLAUDE.md 中自定义压缩行为，如 「压缩时，始终保留完整的已修改文件列表和测试命令」，确保关键 context 在总结后存活
  • 对不需要留在 context 中的快速提问，用 /btw。答案出现在可关闭的浮层中，不进入对话历史，查细节而不增长 context。

用 subagent 做调研

用 「用 subagent 调研 X」 委派研究任务。它们在独立 context 中探索，保持你的主对话干净用于实现。

Context 是你的根本约束，subagent 因此是最强大的工具之一。Claude 调研代码库时会读大量文件，全部消耗你的 context。Subagent 在独立 context window 中运行，只回传摘要：

Use subagents to investigate how our authentication system handles token
refresh, and whether we have any existing OAuth utilities I should reuse.

Subagent 探索代码库、读相关文件、回传发现——全程不污染你的主对话。

实现完成后也可以用 subagent 做验证：

use a subagent to review this code for edge cases

用检查点回退

Claude 的每个操作都会创建检查点（译者注：指 Claude Code 内置的 checkpoint 功能，可回退对话和代码状态到任意先前节点）。你可以将对话、代码或两者恢复到任意先前检查点。

Claude 在做更改前自动创建检查点。双击 Escape 或运行 /rewind 打开回退菜单。你可以只恢复对话、只恢复代码、两者都恢复、或从选定消息开始总结。详见 Checkpointing。

不必谨小慎微地规划每一步。直接让 Claude 尝试有风险的方案——不行就回退，换一种方法。检查点跨会话持久化，关闭终端后仍可回退。

检查点只跟踪 Claude 做出的更改，不跟踪外部进程。它不能替代 git。

恢复对话

运行 claude --continue 接续上次对话，或 --resume 从近期会话中选择。

Claude Code 在本地保存对话。任务跨越多个会话时，无需重新解释上下文：

claude --continue    # 恢复最近的对话
claude --resume      # 从近期对话中选择

用 /rename 给会话起描述性名称，如 "oauth-migration" 或 "debugging-memory-leak"，便于日后查找。把会话当分支用：不同工作流可以有独立的持久 context。

自动化与规模化

一旦你能高效使用一个 Claude，就用并行会话、非交互模式和扇出模式成倍放大产出。

前文假设一人一 Claude 一对话。但 Claude Code 支持水平扩展。本节展示如何做得更多。

运行非交互模式

在 CI、pre-commit hook 或脚本中使用 claude -p "prompt"。加 --output-format stream-json 获取流式 JSON 输出。

用 claude -p "your prompt" 无需会话即可运行 Claude。非交互模式是将 Claude 集成到 CI pipeline、pre-commit hook 或任何自动化工作流的方式。输出格式支持编程解析：纯文本、JSON 或流式 JSON。

# 一次性查询
claude -p "Explain what this project does"

# 结构化输出供脚本使用
claude -p "List all API endpoints" --output-format json

# 流式输出用于实时处理
claude -p "Analyze this log file" --output-format stream-json

运行多个 Claude 会话

并行运行多个 Claude 会话以加速开发、运行隔离实验或启动复杂工作流。

三种主要方式运行并行会话：

  • Claude Code 桌面应用：可视化管理多个本地会话。每个会话有独立的 worktree（译者注：指 git worktree，允许同一仓库同时签出多个工作目录）。
  • Claude Code Web 版：在 Anthropic 安全云基础设施上运行，使用隔离 VM。
  • Agent teams：多个会话的自动化协调，共享任务、消息传递和团队负责人。

并行化之外，多会话还支持以质量为中心的工作流。全新 context 提升代码审查质量——Claude 不会偏向它刚写的代码。

例如，使用 Writer/Reviewer 模式：

会话 A（Writer）会话 B（Reviewer）
Implement a rate limiter for our API endpoints
Review the rate limiter implementation in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns.
Here's the review feedback: [Session B output]. Address these issues.

类似的方式也适用于测试：一个 Claude 写测试，另一个写代码使测试通过。

跨文件扇出

循环调用 claude -p 处理每个任务。用 --allowedTools 限定批量操作的权限范围。

大规模迁移或分析时，可将工作分发到多个并行 Claude 调用：

1. 生成任务列表

让 Claude 列出所有需要迁移的文件（如 「列出所有 2000 个需要迁移的 Python 文件」）。

2. 写脚本循环处理
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
3. 先试几个文件，再全量运行

根据前 2-3 个文件的情况调整 prompt，然后全量运行。--allowedTools 限制 Claude 可做的操作——无人值守运行时这很重要。

也可将 Claude 集成到已有数据/处理 pipeline 中：

claude -p "<your prompt>" --output-format json | your_command

开发阶段用 --verbose 调试，生产环境关闭。

用 auto mode 实现无人值守运行

需要不间断执行且有后台安全检查时，用 auto mode。分类器模型在命令运行前审查，阻止权限升级、未知基础设施操作和恶意内容驱动的操作，同时让常规工作无需确认即可进行。

claude --permission-mode auto -p "fix all lint errors"

对使用 -p 标志的非交互运行，如果分类器反复阻止操作，auto mode 会中止——因为没有用户可以兜底。详见 when auto mode falls back。

避免常见失败模式

以下是常见错误。尽早识别能节省大量时间：

  • 大杂烩会话。你从一个任务开始，中途问了不相关的东西，又回到第一个任务。Context 被无关信息填满。
    解法：在无关任务之间用 /clear。
  • 反复纠正。Claude 做错了，你纠正，还是不对，再纠正。Context 被失败方案污染。
    解法：两次纠正失败后，/clear，用融入所学的更好 prompt 重新开始。
  • 臃肿的 CLAUDE.md。文件太长，Claude 忽略其中一半——重要规则淹没在噪声里。
    解法：狠心修剪。如果 Claude 在没有某条指令的情况下已做得正确，删掉它或转为 hook。
  • 信任-验证缺口。Claude 产出看似合理但未处理边界情况的实现。
    解法：始终提供验证手段（测试、脚本、截图）。如果你没法验证，就别上线。
  • 无限探索。你让 Claude「调研」某东西但没划定范围。Claude 读了几百个文件，context 被塞满。
    解法：缩小调研范围，或用 subagent，让探索不消耗你的主 context。

培养你的直觉

本指南中的模式不是铁律。它们是通用的起点，未必适用于每种情况。

有时你应该让 context 累积——因为你深陷一个复杂问题，历史记录有价值。有时你应该跳过规划让 Claude 自己想办法——因为任务本身就是探索性的。有时模糊 prompt 恰好合适——因为你想先看看 Claude 如何理解问题，再加以约束。

留意什么有效。Claude 产出优秀结果时，注意你做了什么：prompt 结构、提供的上下文、所处的模式。Claude 挣扎时，问为什么——context 太嘈杂？Prompt 太模糊？任务对单次处理来说太大？

假以时日，你会培养出指南无法捕捉的直觉。你会知道何时具体、何时开放，何时规划、何时探索，何时清空 context、何时让它累积。
