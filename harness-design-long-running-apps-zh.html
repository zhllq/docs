长时运行应用开发的 Harness 设计

Harness 设计是 agentic coding 前沿性能的关键。我们如何突破 frontend 设计和长时间自主编程的天花板。

作者：Prithvi Rajasekaran（Anthropic Labs 成员）

过去几个月，我一直在攻克两个相互关联的难题：让 Claude 生成高质量的 frontend 设计，以及让它在无人干预的情况下独立完成完整应用的开发。这项工作脱胎于我们早期对 frontend design skill 和 long-running coding agent harness 的探索——通过 prompt engineering 和 harness 设计，Claude 的表现已大幅超过基线，但两者都遭遇了性能瓶颈。

为了突破瓶颈，我从 Generative Adversarial Networks（GAN）中获得灵感，设计了一套由 generator agent 和 evaluator agent 构成的 multi-agent 架构。要让 evaluator 的打分既可靠又有品位，首先需要把"这个设计好看吗？"这样的主观判断转化为具体可评分的标准。

随后，我把这些方法迁移到长时间自主编程任务上，并延续了早期 harness 工作中总结出的两条经验：把构建过程拆解成可管理的小块，以及使用结构化 artifact 在 session 之间传递上下文。最终形成了一套 planner → generator → evaluator 的三 agent 架构，能够在数小时的自主编程会话中交付功能完整的 full-stack 应用。

朴素实现为何行不通

我们已经证明，harness 设计对 long-running agentic coding 的效果影响显著。在早期实验中，我们用一个 initializer agent 将产品 spec 分解成任务列表，再由 coding agent 逐 feature 实现，并通过传递 artifact 在 session 之间衔接上下文。更广泛的开发者社区也得出了类似结论——比如 "Ralph Wiggum" 方法，通过 hook 或脚本让 agent 持续循环迭代。

但有些问题始终存在。对于更复杂的任务，agent 仍会随着时间推移逐渐偏轨。我们观察到两种典型的失败模式：

失败模式一：Context 退化

随着 context window 被填满，模型在长任务中会逐渐失去连贯性（详见我们关于 context engineering 的文章）。部分模型还会出现 "context anxiety"（上下文焦虑）——在接近其感知到的 context 上限时过早收尾。

Context reset（清空 context window、启动全新 agent，并通过结构化 handoff 传递前一 agent 的状态和后续步骤）能够同时解决这两个问题。

这与 compaction 不同——compaction 会在原地压缩较早的对话，让同一个 agent 在缩短的历史上继续工作。compaction 保留了连续性，但无法给 agent 一块干净的石板，context anxiety 仍可能持续。Context reset 提供的是真正的"重置"，代价是 handoff artifact 必须携带足够的状态，让下一个 agent 能无缝接力。在早期测试中，Claude Sonnet 4.5 表现出较强的 context anxiety，仅靠 compaction 无法支撑强大的长任务性能，因此 context reset 成为 harness 设计的核心机制。

失败模式二：自我评估偏差

当让 agent 对自己生成的内容评分时，它们往往会给出充满信心的称赞——即便在人类观察者眼中，质量明显平庸。这个问题在设计等主观任务上尤为突出，因为没有像软件测试那样的二值化验证标准。

解决方案是把执行工作的 agent 和判断质量的 agent 分离开来。单纯分离并不能立刻消除宽容倾向——evaluator 本身也是 LLM，同样倾向于对 LLM 输出宽大处理。但将一个独立的 evaluator 调教成持怀疑态度，远比让 generator 自我批判容易得多。一旦外部反馈存在，generator 就有了具体可迭代的依据。

Frontend 设计：让主观质量变得可评分

图：frontend 设计的迭代过程示例

我先从 frontend 设计入手，因为这里的自我评估问题最为明显。在没有任何干预的情况下，Claude 倾向于生成技术上可行但视觉上毫无亮点的安全布局。

两个核心洞察塑造了我的 harness 设计。第一，美学虽然无法完全量化，但可以通过编码了设计原则和偏好的评分标准来改善。"这个设计漂亮吗？"难以一致回答，但"这个设计符合我们的设计原则吗？"就给了 Claude 可以具体打分的依据。第二，将 frontend 生成和 frontend 评分分离，就能创建一个不断驱动 generator 提升输出质量的反馈回路。

四维评分标准

我制定了以下四项评分标准，同时写入 generator 和 evaluator 的 prompt：

Design Quality

设计整体性

颜色、字体、布局、图像等各要素是否融为一体，形成清晰的视觉风格和氛围？

Originality

原创性

是否有自定义决策的痕迹，而非模板布局和 AI 的惯用套路？优质作品应让人感受到刻意的创意选择。

Craft

工艺执行

字体层级、间距一致性、色彩和谐、对比度——这是技术能力的检验，而非创意检验。

Functionality

功能可用性

用户能否理解界面用途、找到主要操作并完成任务，与美观无关？

我有意将 Design Quality 和 Originality 的权重设得高于 Craft 和 Functionality——后两者 Claude 默认就能做得不错，而前两者则经常产出平平无奇的作品。标准明确惩罚了那些高度通用的"AI 味"模式，倒逼模型进行更大胆的审美冒险。

我用 few-shot 示例对 evaluator 进行了校准，确保其判断与我的偏好对齐，也减少了迭代过程中的分数漂移。

Harness 的工作方式

整个系统基于 Claude Agent SDK 构建。Generator agent 根据用户 prompt 生成一个 HTML/CSS/JS frontend，evaluator 获得 Playwright MCP，可以直接与运行中的页面交互，逐项打分并写出详细评语。实际上，evaluator 会自主导航页面，截图后仔细研读实现，再给出评估结果。该反馈作为下一次迭代的输入传回 generator。

每次生成运行 5–15 次迭代。每次评估后，generator 还需做出策略决策：若分数走势良好则继续深化当前方向；若整体思路效果不佳则彻底转换风格。由于 evaluator 是在实时导航页面而非给静态截图打分，每个周期都需要真实的时钟时间，完整运行最长可达四小时。

图：多轮迭代后的 frontend 设计演变
值得一提的案例：我让模型为一家荷兰艺术博物馆创建网站。第九次迭代时，它生成了一个干净的深色调主页。然后，第十次迭代发生了戏剧性转变——它彻底推翻原有方案，将网站重新构想为一个空间体验：用 CSS perspective 渲染出一个棋盘格地板的 3D 房间，艺术品以自由排布的方式挂在墙上，并通过"门道"式导航在画廊间穿行，而非滚动或点击。这是我此前从单次生成中从未见过的创意跳跃。

跨多次运行来看，evaluator 的评分总体上随迭代提升后趋于平稳，但仍有提升空间。值得注意的是，标准中的措辞——比如"最优秀的设计是博物馆级别的"——会以我未完全预料到的方式引导 generator，说明与标准相关的 prompting 语言直接塑造了输出的风格。

扩展到 Full-Stack 编程

有了上述发现，我将这套 GAN 启发的模式应用到 full-stack 开发。Generator–evaluator 回路天然契合软件开发生命周期——代码评审和 QA 在结构上起着与 design evaluator 相同的作用。

三 Agent 架构

🗺️

Planner

接收 1–4 句话的简短 prompt，将其扩展为完整的产品 spec。Planner 被要求保持宏大的产品愿景，聚焦于产品背景和高层技术设计，而非细粒度实现细节。这样做是为了避免 planner 提前锁定错误的技术细节，导致错误向下游级联。同时，planner 还会主动寻找机会，将 AI feature 织入产品 spec 中。

↓
⚙️

Generator

以 sprint 方式工作，每次从 spec 中取一个 feature 推进实现。技术栈为 React / Vite / FastAPI / SQLite（后期升级为 PostgreSQL）。每个 sprint 结束后，generator 先自我评估，再将工作移交 QA。它还使用 git 进行版本控制。

↓
🔍

Evaluator

使用 Playwright MCP 像真实用户一样点击运行中的应用，测试 UI feature、API endpoint 和数据库状态。随后根据发现的 bug 和预设标准（覆盖产品深度、功能性、视觉设计、代码质量）对每个 sprint 打分。任一维度低于阈值，sprint 即失败，generator 收到详细反馈后重新开始。

Sprint Contract 机制

每个 sprint 开始前，generator 和 evaluator 先协商 sprint contract：在写任何代码之前，先就"这个 sprint 做到什么程度算完成"达成一致。Generator 提出将要构建的内容及验收标准，evaluator 审查该提案，两者反复协商直到达成共识。这一步桥接了高层 spec 与可测试实现之间的鸿沟。

两个 agent 通过文件进行通信：一个 agent 写文件，另一个读取并在该文件或新文件中回应，确实拿到 上一个的输出后再继续。这保持了工作对 spec 的忠实，同时避免了过早锁定实现细节。

Solo 版本的问题

初打开 Solo 版本，界面看上去符合预期。但随着操作深入，问题接连出现：布局浪费空间，固定高度的面板让大量视口空白；工作流程僵硬——填充关卡之前需要先创建精灵和实体，但界面没有任何引导。更关键的是，游戏核心功能是坏的：实体出现在屏幕上，但对任何输入都没有响应。翻看代码才发现，实体定义和游戏运行时之间的连接断裂了，而界面上没有任何错误提示。

Full Harness 版本的表现

Harness 版本从同一句 prompt 出发，但 planner 将其扩展为覆盖 16 个 feature、分为 10 个 sprint 的详细 spec。除了核心编辑器和游戏模式，spec 还包括精灵动画系统、行为模板、音效与音乐、AI 辅助的精灵生成和关卡设计，以及支持分享链接的游戏导出。Planner 还读取了我们的 frontend design skill，并将视觉设计语言纳入 spec。

图：harness 版本生成的游戏制作器界面

应用的打磨程度和流畅感立刻超越了 Solo 版本：画布铺满整个视口、面板尺寸合理、界面有统一的视觉风格。Sprite editor 更为丰富，工具面板更整洁，色彩选择器和缩放控件都更易用。由于我要求 planner 将 AI feature 融入 spec，应用还内置了 Claude 集成，允许用户通过自然语言 prompt 生成游戏各个部分，大幅提升工作流效率。

最大的差异在于游戏模式——我能够真正移动角色并玩这个游戏。物理有一些粗糙之处（角色跳上平台后出现重叠感），但核心功能是工作的，而 Solo 版本完全做不到这一点。

精简 Harness

第一版 harness 结果令人鼓舞，但它臃肿、缓慢且昂贵。下一步的逻辑是：在不降低性能的前提下简化 harness。

核心原则：harness 的每个组件都编码了一个关于"模型自身无法做到什么"的假设。这些假设值得不断检验——因为它们可能本来就是错的，也可能随着模型的进步迅速过时。我们在 Building Effective Agents 中提到过这个思路："找到最简单的可行方案，只在必要时才增加复杂度。"

在第一次简化尝试中，我砍掉了大量组件并尝试了一些新想法，但无法复现原来的性能。更糟的是，很难判断 harness 中哪些部分是真正起作用的。于是我转向更严格的方法：每次只移除一个组件，观察对最终结果的影响。

此时 Opus 4.6 的发布提供了进一步简化的动力。发布公告指出：Opus 4.6 "规划更周全，能更持久地执行 agentic 任务，在更大型代码库中运行更可靠，且具备更好的代码审查和调试能力来发现自身错误"，同时在 long-context retrieval 上也有大幅提升——这些正是 harness 一直在弥补的能力。

移除 Sprint 结构

首先移除了整个 sprint 结构。Sprint 的意义在于将工作分解成模型能连贯处理的小块。鉴于 Opus 4.6 的提升，有理由相信模型可以在无需这种分解的情况下独立完成工作。

Planner 和 evaluator 都保留了，因为两者仍有明显价值。没有 planner，generator 会缩小范围——直接拿到原始 prompt 后，它会不先规划就开始构建，最终生成的应用不如 planner 展开后的版本功能丰富。

移除 sprint 结构后，evaluator 也从每个 sprint 后打分改为最终统一跑一次。这改变了 evaluator 对不同任务的价值权重：对于模型能独立处理的任务，evaluator 变成了不必要的开销；但对于仍处于模型能力边界的部分，evaluator 依然能带来真实收益。

Evaluator 不是一个是/否的固定决策。当任务超出当前模型独立完成的可靠范围时，evaluator 才值得付出成本。

在结构简化的同时，我也加强了 prompt，让 harness 更好地将 AI feature 融入每个应用——具体来说，是让 generator 构建一个真正的 agent，通过工具驱动应用自身的功能。这需要大量调试，因为相关知识在 Claude 的训练数据中覆盖较薄。但经过足够的调优，generator 开始能正确构建 agent 了。

QA 仍然捕获了真实问题

第一轮 QA 反馈：

"这是一个强大的应用，设计保真度出色，AI agent 稳健，backend 优良。主要问题是 Feature Completeness——应用看起来令人印象深刻，AI 集成也运行良好，但核心 DAW feature 很多是只展示、无交互的：clip 无法在 timeline 上拖移，没有乐器 UI 面板（合成器旋钮、鼓垫），也没有视觉效果编辑器（EQ 曲线、压缩器电平表）。这些不是边缘情况，而是让 DAW 真正可用的核心交互，spec 中明确要求了它们。"

未来展望

随着模型持续进步，我们大致可以预期它们能处理时间更长、任务更复杂的工作。在某些情况下，这意味着围绕模型的 scaffold 会随着时间推移变得不那么重要，开发者只需等待下一个模型迭代，某些问题便会自行解决。但反过来，模型越强大，就越有空间去开发能实现更复杂任务的 harness——这些任务是模型在 baseline 状态下无法完成的。

从这项工作中，我总结出几点值得延续的经验：

01

持续实验与 trace 阅读

用你正在构建的目标模型做实验，在真实问题上读它的 trace，并调优性能以达到预期目标。

02

任务分解 + 专项 Agent

对更复杂的任务，将其分解并为每个方面应用专门的 agent，往往能释放额外性能空间。

03

新模型上线即重审 Harness

新模型落地后，系统性地审查 harness，剥离不再 load-bearing 的部分，添加新能力以达到此前不可能实现的目标。

04

设计空间在移动，不在收缩

随着模型进步，有趣的 harness 组合空间不会缩小，而是在移动。AI 工程师的价值在于持续找到下一个新颖的组合。

RetroForge — 2D 复古游戏制作器

产品概述

RetroForge 是一个基于浏览器的创意工作室，用于设计和构建 2D 复古风格电子游戏。它将经典 8-bit 和 16-bit 游戏美学的怀旧魅力与现代直观的编辑工具结合——让任何人都能在不编写传统代码的情况下，将游戏创意变为现实。

平台提供四个集成创意模块：基于 tile 的 Level Editor（关卡编辑器）、用于制作像素艺术的 Sprite Editor（精灵编辑器）、可视化的 Entity Behavior（实体行为）系统，以及 Playable Test Mode（即时游戏测试模式）。通过在整个流程中织入 AI 辅助（由 Claude 驱动），RetroForge 极大加速了创作过程——帮助用户通过自然语言交互生成精灵、设计关卡和配置行为。

功能概览（部分）

  • 项目管理仪表板：创建、浏览、复制、删除项目
  • Tile 化关卡编辑器，支持多图层设计
  • 像素艺术 Sprite 编辑器，含动画系统
  • 可视化 Entity Behavior 模板配置
  • 即时游戏测试模式
  • AI 辅助精灵生成与关卡设计（通过 Claude）
  • 游戏导出与分享链接
  • 音效与背景音乐支持

技术要点总结

本节梳理文章中所有关键技术点，方便快速回顾。

核心问题与动机

  • Long-running agentic coding 存在两大失败模式：context window 填满导致模型失去连贯性（context degradation）；以及 agent 对自身输出过度乐观（self-evaluation bias）。
  • 某些模型（如 Sonnet 4.5）会出现 context anxiety，在接近感知的 context 上限时过早结束工作。
  • compaction（原地压缩历史对话）无法彻底解决 context anxiety，只有 context reset（完全清空并由 structured handoff 衔接）才能给 agent 一块干净的石板。

核心架构：GAN 启发的 Multi-Agent 设计

  • 受 Generative Adversarial Networks (GAN) 启发，设计 generator agent + evaluator agent 的对抗式协作结构。
  • Full-stack 版本扩展为三 agent：Planner → Generator → Evaluator。
  • 使用 Claude Agent SDK 管理 orchestration，Playwright MCP 赋予 evaluator 与运行中应用实时交互的能力。

Frontend Design 评分体系

  • 四维可量化标准：Design Quality、Originality、Craft、Functionality。
  • 标准权重：Design Quality 和 Originality 高于 Craft 和 Functionality，以打压通用 "AI 味" 输出。
  • Evaluator 用 few-shot 示例进行校准，减少分数漂移。
  • 每轮迭代 evaluator 实时导航页面（非静态截图打分），完整 run 可达 5–15 次迭代、耗时最长 4 小时。

Full-Stack Coding Harness 机制

  • 技术栈：React / Vite / FastAPI / SQLite（后升级为 PostgreSQL）+ git 版本控制。
  • Sprint 结构：generator 每次处理一个 feature，sprint 结束前先自评再移交 QA。
  • Sprint Contract：generator 和 evaluator 在写代码前协商"完成"标准，桥接高层 spec 与可测试实现。
  • Agent 间通信通过文件完成，避免直接耦合。
  • Evaluator 设置硬性通过阈值（任意维度低于阈值则 sprint 失败），强制 generator 修复后重来。

关键效果对比

  • 复古游戏制作器：Solo ($9 / 20 分钟) vs Full Harness ($200 / 6 小时)。Solo 版本核心游戏功能损坏；harness 版本功能完整，含 AI 集成。
  • DAW（V2 简化 Harness）：3 小时 50 分钟，$124.70，交付可工作的 timeline、mixer、transport 及 AI agent 驱动的作曲功能。

Harness 演进：随模型能力动态调整

  • Opus 4.6 具备更强的 long-context retrieval、更稳定的 agentic 执行，使得之前必要的 context reset 和 sprint decomposition 可以去除。
  • 移除 sprint 结构后，evaluator 改为单次终态 QA，仍能捕获深层嵌套 feature 中的漏网 bug。
  • Evaluator 的价值不是固定的——当任务超出模型独立可靠完成的边界时才值得引入；该边界随模型能力向外移动。
  • 调优 evaluator 是迭代过程：读日志 → 找判断分歧 → 更新 QA prompt，需多轮循环。

核心设计原则

  • Separation of Concerns 驱动质量：专门的 evaluator 远比 generator 自评更有效，即便 evaluator 本身不完美。
  • Criteria formalization：将主观判断（"好看吗？"）转化为可量化标准，才能产生可迭代的反馈。
  • 假设检验习惯：harness 的每个组件都是一个关于模型能力边界的假设，应随新模型发布而重新检验。
  • 设计空间在移动：模型进步不会缩小 harness 设计的有趣空间，而是将其向更复杂的任务方向推移。
